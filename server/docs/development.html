Development Environment

Setting up your development environment for contributing to the Roblox Multi-Game System.

Prerequisites

Required Software
  • Docker: Version 20.10 or higher
  • Docker Compose: Version 2.0 or higher
  • Git: Version control system
  • Node.js: Version 18 or higher for development tools
  • NPM: Node Package Manager
Recommended Tools
  • Editor: VS Code with Docker extension
  • Terminal: Integrated terminal in editor
  • Browser: Chrome/Firefox for debugging
  • API Client: Postman or Insomnia
System Requirements
  • OS: Linux, macOS, or Windows with WSL2
  • CPU: 4+ cores recommended
  • RAM: 8GB minimum, 16GB recommended
  • Disk: 50GB free space

Setup Instructions

1
Clone the Repository
# Clone the repository git clone <repository-url> cd rb-server/server # Set up SSH keys for Git (recommended) ssh-keygen -t rsa -b 4096 -C "your_email@example.com" ssh-add ~/.ssh/id_rsa
2
Configure Environment
# Copy environment file cp .env.example .env # Edit environment variables (see .env.example for descriptions) nano .env # For development, you might want to override these: export DB_PASSWORD=dev_password export REDIS_PASSWORD=dev_redis_password export JWT_SECRET=dev_secret_key
3
Start Development Environment
# Build and start all services for development docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d # Or use the development script if available ./start.sh start-dev # Build services (if you made changes) docker compose build # Start services docker compose up -d
4
Verify Setup
# Check that all services are running docker compose ps # View logs to make sure services are healthy docker compose logs # Access the system in your browser # Web Interface: http://localhost:8081 # API: http://localhost:8081/api/health

Development Workflow

Step-by-step process for developing new features and fixing issues.

General Development Workflow

1
Create Feature Branch

Always create your work in a branch based on the main branch:

# Update your local main branch git checkout main git pull origin main # Create a feature branch git checkout -b feature/my-new-feature # OR git checkout -b fix/issue-description
2
Make Changes

Follow coding standards and write clean, maintainable code:

  • Use meaningful variable and function names
  • Write concise, informative comments
  • Follow existing code style and patterns
  • Write unit tests for new functionality
  • Update documentation as needed
// Example of good function documentation /** * Calculate player score based on actions * @param {Object} player - Player object with stats * @param {Array} actions - Array of actions performed * @returns {number} Calculated score */ function calculatePlayerScore(player, actions) { // Implementation here return score; }
3
Test Locally

Run comprehensive local tests:

# Run tests in development environment ./start.sh test # Run specific service tests docker compose exec roblox-server npm test # Check for security vulnerabilities npm audit # Lint code npm run lint
4
Commit Changes

Write clear, descriptive commit messages:

# Format: type(scope): brief description git add . git commit -m "feat(game-manager): add player score calculation" # Or for fixes: git commit -m "fix(game-logic): resolve race condition in player movement"
5
Push and Create PR
# Push your branch to remote git push origin feature/my-new-feature # Create a pull request through GitHub UI # - Provide detailed description # - Link related issues # - Tag reviewers

Code Review Guidelines

When Reviewing Others' Code
  • Check for security vulnerabilities
  • Verify input validation and sanitization
  • Assess performance implications
  • Ensure proper error handling
  • Confirm code follows established patterns
  • Verify tests cover edge cases
When Having Your Code Reviewed
  • Provide context in the PR description
  • Link to related issues or designs
  • Explain the problem being solved
  • Highlight any potential concerns
  • Respond promptly to review comments

Game Development

Creating and integrating new games into the multi-game system.

Game Development Process

1
Create Game Directory

Set up the directory structure for your new game:

mkdir games/my-awesome-game cd games/my-awesome-game # Create the standard structure mkdir assets modules scripts config touch README.md game.json Dockerfile
2
Define Game Metadata

Create a game.json file with game information:

{ "id": "my-awesome-game", "name": "My Awesome Game", "version": "1.0.0", "description": "A description of the game", "author": "Your Name", "category": "puzzle", // strategy, puzzle, adventure, etc. "tags": ["multiplayer", "puzzle", "casual"], "maxPlayers": 4, "minPlayers": 1, "estimatedDuration": 30, "languages": ["en", "de"], "settings": { "defaultMap": "default-map", "sessionTimeout": 3600, "gameRules": { // Custom game-specific rules } }, "dependencies": [], "compatibility": { "minSystemVersion": "1.0.0", "maxSystemVersion": "2.0.0" } }
3
Develop Game Logic

Create the main game logic in scripts/main.lua:

local game = {} function game.init() print("Initializing My Awesome Game...") -- Setup game state game.state = { players = {}, assets = {}, settings = json.load("/app/config/settings.json"), events = EventQueue:new() } return game.state end function game.startSession(sessionConfig) local session = { id = generateId(), startTime = os.time(), settings = sessionConfig, players = {}, state = "active" } -- Notify subscribers game.state.events:push("game_started", session) return session end function game.processEvent(event) if event.type == "player_join" then return game.handlePlayerJoin(event.data) elseif event.type == "player_action" then return game.handlePlayerAction(event.data) end return false end function game.handlePlayerJoin(data) local player = { id = data.playerId, name = data.name, position = data.position, joinedTime = os.time() } game.state.players[player.id] = player game.state.events:push("player_joined", player) return player end function game.handlePlayerAction(data) local player = game.state.players[data.playerId] if not player then return {error = "Player not found"} end -- Process action based on type if data.action == "move" then player.position = data.position elseif data.action == "interact" then -- Handle interaction end game.state.events:push("player_action", { playerId = data.playerId, action = data.action, position = player.position }) return {success = true} end return game
4
Create Docker Configuration

Create a Dockerfile for your game:

FROM node:18-alpine # Install Lua and dependencies RUN apk add --no-cache \ lua5.3 \ lua5.3-dev \ luarocks \ curl \ git # Install Lua modules - adjust as needed RUN luarocks install lua-cjson RUN luarocks install luasocket WORKDIR /app # Copy game files COPY . . # Install Node.js packages if needed RUN npm install # Expose game port (adjust as needed) EXPOSE 27015 # Health check endpoint (adjust path as needed) HEALTHCHECK --interval=30s --timeout=10s --start-period=40s --retries=3 \ CMD curl -f http://localhost:8080/health || exit 1 # Start the game CMD ["node", "server.js"]
5
Test Your Game

Thoroughly test your game before integration:

# Build your game container docker build -t game-my-awesome-game:1.0.0 . # Test in isolation docker run --rm -p 27015:27015 game-my-awesome-game:1.0.0 # Run game-specific tests ./scripts/test-game.sh my-awesome-game # Run multiplayer simulation ./scripts/test-multiclient.sh --game=my-awesome-game --clients=4
6
Register with Game Manager

Integrate your game with the game manager:

# Register your game with the system curl -X POST http://localhost:8081/api/games \ -H "Content-Type: application/json" \ -H "Authorization: Bearer YOUR_JWT_TOKEN" \ -d '{ "name": "My Awesome Game", "version": "1.0.0", "description": "A description of the game", "category": "puzzle", "maxPlayers": 4 }'

Game Asset Guidelines

Image Assets
  • Format: Use WebP for photographs, PNG for graphics
  • Compression: Optimize with tools like ImageOptim
  • Size: Keep below 1MB per image when possible
  • Resolution: Provide multiple resolutions for different devices
Audio Assets
  • Format: Use OGG (Vorbis) for music, WAV for SFX
  • Compression: Use VBR (Variable Bitrate) at 128-256 kbps
  • Volume: Normalize to ensure balanced audio
Model Assets
  • Format: Use OBJ or GLTF for 3D models
  • Polygons: Keep models low-poly (under 10k triangles)
  • UV Mapping: Ensure proper UV unwrapping

API Documentation

Complete API reference for system and game interactions.

Authentication API

Login
POST /api/auth/login Content-Type: application/json { "username": "string (required)", "password": "string (required)" } Response: { "token": "jwt_token_string", "expiresIn": 86400, "user": { "id": "string", "username": "string", "role": "string", "permissions": ["string"] } }
Token Validation
GET /api/auth/verify Authorization: Bearer YOUR_JWT_TOKEN Response: { "valid": true, "user": { "id": "string", "username": "string", "role": "string" }, "expiresIn": 86300 }

Game Management API

Create Game
POST /api/games Authorization: Bearer YOUR_JWT_TOKEN Content-Type: application/json { "name": "My Game", "version": "1.0.0", "description": "Game description", "author": "Author Name", "category": "strategy", "maxPlayers": 8, "tags": ["multiplayer", "strategy"], "settings": { "defaultMap": "default-map", "timeLimit": 1800 } }
Get Game Details
GET /api/games/{gameId} Authorization: Bearer YOUR_JWT_TOKEN Response: { "game": { "id": "string", "name": "string", "version": "string", "description": "string", "author": "string", "category": "string", "tags": ["string"], "maxPlayers": "number", "settings": "object", "status": "string" }, "instance": { // Current instance info if running } }
Start Game Instance
POST /api/games/{gameId}/start Authorization: Bearer YOUR_JWT_TOKEN Content-Type: application/json { "maxPlayers": 6, "map": "desert-battlefield", "difficulty": "normal", "settings": { "resourcesMultiplier": 1.0, "buildTimeMultiplier": 1.0 } }

Real-Time Communication

WebSocket Connection
const socket = io('ws://localhost:8080', { withCredentials: true, extraHeaders: { "Authorization": "Bearer YOUR_JWT_TOKEN" } }); socket.on('connect', () => { console.log('Connected to game server'); }); socket.on('games_list', (data) => { console.log('Available games:', data.games); }); socket.on('player_joined', (player) => { console.log('New player:', player.username); }); socket.emit('player_action', { action: 'move', position: {x: 10, y: 20, z: 5} });

Deployment

Processes and tools for deploying to different environments.

Deployment Environments

Development
  • Local Docker environment
  • Hot reloading enabled
  • Verbose logging
  • Less security restrictions
Staging
  • Production-like environment
  • Security features enabled
  • Performance monitoring
  • Feature flag testing
Production
  • Full security protocols
  • Scalability features
  • Disaster recovery
  • Monitoring and alerts

Deployment Commands

Development Deployment
# Start development environment ./start.sh start-dev # Or manually docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d # View development logs ./start.sh logs -f
Production Deployment
# Full production deployment ./start.sh start-prod # Or with specific compose files docker compose -f docker-compose.prod.yml -f docker-compose.secure.yml up -d # With monitoring docker compose -f docker-compose.prod.yml --profile monitoring up -d # Blue-green deployment docker compose -f docker-compose.blue.yml up -d
Rollback Procedure
# Check previous deployment docker images # Rollback to previous version docker compose down docker compose -f docker-compose.prod.yml up -d # Or rollback specific service docker compose down roblox-server docker compose up -d roblox-server --build

During Deployment
  • Monitor deployment progress

After Deployment
  • Run post-deployment tests
  • Monitor application performance

Contributing

Guidelines for contributing to the project as a team member.

Community Guidelines

Always follow respectful and inclusive communication. Ensure all contributions benefit the community and maintain code quality standards.

Code Standards

JavaScript/Node.js
  • Use ES6+ features appropriately
  • Follow Airbnb JavaScript style guide
  • Use JSDoc for complex functions
  • Prefer const/let over var
  • Use camelCase for variables/functions
  • Use PascalCase for classes/constructors
Lua
  • Use snake_case for variables/functions
  • Prefer local variables over globals
  • Comment complex algorithms
  • Follow common Lua patterns and idioms
Shell Scripts
  • Use POSIX-compatible syntax
  • Include proper error handling
  • Document arguments and usage
  • Use consistent indentation (2 spaces)

Documentation Standards

Code Comments
  • Explain why, not what the code does
  • Document complex algorithms and business logic
  • Mark temporary solutions with TODO comments
  • Reference related issues or tickets
External Documentation
  • Update README files when changing public interfaces
  • Create or update API documentation
  • Provide examples for new features
  • Include troubleshooting information
